# Benchmark and profiling results, 2026-06-12

Environment: Windows 11, JDK 21.0.10, one JMH thread.
All setup that builds a corpus or index is performed at `Level.Trial`.

## Boolean operators

Sources:

- `jmh-boolean-core-2026-06-12.json`: 4 x 500 ms warmup, 5 x 500 ms
  measurement, 2 forks;
- `jmh-operators-2026-06-12.json`: positional and difference operators,
  2 x 300 ms warmup, 3 x 300 ms measurement, 2 forks.

Every operation within a column uses the same two posting lists and controlled
overlap.

| Operation, us/op | rare-rare | rare-medium | rare-common | medium-medium | common-common |
| --- | ---: | ---: | ---: | ---: | ---: |
| adaptive AND | 2.496 | 21.289 | 101.264 | 124.322 | 1281.261 |
| AND without skips | 2.537 | 65.838 | 643.575 | 126.187 | 1273.796 |
| OR | 3.892 | 117.685 | 1084.814 | 202.033 | 1830.012 |
| AND NOT | 3.044 | 20.304 | 128.809 | 149.208 | 1523.565 |
| ADJ | 4.664 | 23.265 | 100.251 | 243.057 | 2961.796 |
| NEAR/3 | 5.457 | 22.159 | 103.099 | 255.856 | 3216.175 |

For skewed lists, skips reduce AND latency by 3.1x for `rare-medium` and
6.4x for `rare-common`. For comparable lists the adaptive implementation
selects the linear path, so skip overhead does not make AND slower than OR.
The JSON files contain JMH errors and confidence intervals.

## Compression size

Synthetic index: 10 000 documents, 80 tokens per document.

| Mode | Payload, bytes | Raw/payload | Full index, bytes |
| --- | ---: | ---: | ---: |
| none | 8 209 548 | 0.994 | 9 467 069 |
| delta | 8 209 548 | 0.994 | 9 467 076 |
| varbyte | 2 676 901 | 3.048 | 3 935 329 |
| delta-varbyte | 2 084 521 | 3.915 | 4 753 236 |
| bitpacking | 1 938 169 | 4.210 | 3 195 213 |
| delta-bitpacking | 1 369 746 | 5.957 | 2 626 725 |
| pfor | 2 294 914 | 3.556 | 3 552 880 |
| delta-pfor | 1 643 321 | 4.966 | 2 902 354 |

`none` and `delta` are slightly larger than the raw integer payload because
each encoded stream stores its element count. Delta alone changes values but
does not reduce 32-bit storage. The full `delta-varbyte` index includes
persistent skip offsets, trading space for streaming `advanceTo`.

The diagnostic output now prints sample docIDs, docID deltas, TF values,
positions, position deltas and the encoded size for every codec.

## Search by codec

Source: `jmh-codec-search-2026-06-12.json`, 4 x 500 ms warmup,
5 x 500 ms measurement, 2 forks.

| Mode | mmap search, us/op |
| --- | ---: |
| none | 4822.808 |
| delta | 4967.043 |
| varbyte | 5026.235 |
| delta-varbyte | 3951.240 |
| bitpacking | 5587.193 |
| delta-bitpacking | 4676.232 |
| pfor | 5972.201 |
| delta-pfor | 5313.103 |

`delta-varbyte` has the best end-to-end query latency because it uses
streaming decode and persistent skip offsets. This does not mean varbyte
decoding is intrinsically faster than uncompressed reads: the standalone
decode benchmark reports `322.484 us/op` for `delta-varbyte` and
`97.203 us/op` for `none`.

The search benchmark exercises different cursor implementations.
`delta-varbyte` uses `DiskPostingCursor`, decodes values on demand and seeks
through stored byte offsets. Other codecs, including `none`, currently decode
and materialize the complete requested posting list before query execution.
Therefore `3951.240` vs `4822.808 us/op` describes the current end-to-end
implementation for this query, not a general property of compression.

for `история AND россия`, `ADJ`, `NEAR/1`,
